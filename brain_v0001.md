# FlarePocket Brain
**Version: v0001**
最終更新: 2026-03-30

---

## 1. プロジェクト概要

**FlarePocket** は日本語YouTube動画（主にビジネス系教育コンテンツ）のテロップ・演出を自動添削するシステム。

- 動画を1秒ごとにフレーム（JPEG）に分解し、AIがフレームを読み込みながらテロップ等の問題を指摘する
- チェック対象：誤字・固有名詞表記・口語表現・全角数字・改行位置・モザイク漏れ・バナー・演出
- 出力：`output/backlog/{動画タイトル}/review_report.txt`

## 2. チェックルール（カテゴリ一覧）
manual.md参照
| コード | カテゴリ | 内容 |
|--------|----------|------|
| **HY1** | 誤字・固有名詞 | 誤字脱字、固有名詞の誤表記（下記参照） |
| **HY2** | 口語・語彙ミス | 二重否定、二重敬語、「やつ」等の口語、助詞抜け、ら抜き言葉 |
| **HY4** | 全角数字→半角 | テロップ内の全角数字（１２３など）を半角に修正 |
| **HY6** | 改行・短テロップ | 助詞直後改行、接続助詞の途中改行、短すぎる1行テロップ |
| **HY7** | 行頭「って」 | 行頭・テロップ冒頭が「って」で始まる場合は「という」に修正 |
| **MO** | モザイク漏れ | ASIN/セラーID/仕入価格/個人情報の露出 |
| **SE** | 演出テロップ | 迫力系・強調テロップなし（音声SEはフレームでは判定不可→指摘しない） |
| **CA** | バナー・ロゴ | バナー未表示、ロゴ欠け、エンドカード誤字 |
| **TX2** | カット候補 | 相槌のみ（「うん」「なるほど」単独）、話途中カット |

### 固有名詞・正式表記チェックリスト
- YouTube（大文字T）
- Amazon（「アマゾン」不可）
- Payoneer（「ペイオニア」不可）
- Yahoo!ショッピング（「！」込みが正式）
- S&P500（500まで含める）
- WEBセミナー（「ウェブセミナー」不可）
- eBay / Shopee / SHOPEE / Sekai Pocket / Zoom / SLS / BUYMA / ASIN
- バレエ（「バレー」は誤り）
- スムーズ（「スムース」は誤り）
- アベニュー・モンテーニュ（「アベニモンターニュ」は誤り）

### MOルールの例外（指摘不要）
- バーコード・商品価格 → 指摘不要（編集D確認済み）
- 実践者の顔写真 → 指摘不要（同意取得済み前提）
- 受講生の成果数字（実名なし）→ 基本指摘不要

## 3. 添削方針

### 厳しさ
**「疑わしければ指摘する」** 方針。指摘数が多くなって構わない。見逃しの方が問題。

- 迷ったら指摘する（省かない）
- 誤字・表記統一・改行位置・1行テロップは量が多く出て当然
- 「演出が足りないかも」「スケールが小さいかも」なども積極的に出す

### 出力形式（必須・この形式以外は不可）
```
- MM:SS HY1 テロップ「xxx」→「yyy」に修正
- MM:SS HY2 テロップ「xxx」→口語表現。「yyy」に修正
- MM:SS CA バナーが未表示。確認してください
```

- 時刻は `MM:SS` または `H:MM:SS`
- 表（テーブル）は使わない、箇条書きのみ
- カテゴリコードを必ず付ける
- HY6（改行位置）の指摘は「意味の切れ目で修正」等の曖昧表現は不可。修正後のテロップ全文を `→「1行目／2行目」` の形式で必ず示す

## 4. 絶対に守るルール（幻覚防止・タイムコード精度）

### ① フレームを見ずに指摘しない
- 「このシーンにこのテロップがあるはず」と推測して指摘しない
- 指摘前に必ず `read_frame` で該当フレームを目視確認する
- フレームを確認して「確かに存在する」と確認できた内容のみ指摘する
- 疑わしい場合は前後3〜5フレームも確認してから判断する

### ② タイムコードは計算・推測しない
- `read_frame` が返したファイル名の時刻をそのまま使う
  - 例：`frame_01346.jpg` → タイムコードは `22:26`（1346÷60=22余26）
- 自分でフレーム番号から時刻を計算して推測しない
- テロップが最初に出現するフレームを特定してから記載する

### ③ 疑問文「？」は必ずフレームで確認
- テロップに「？」がないと思っても、実際のフレームを開いて目視確認してから指摘する
- script.txt（自動生成字幕）は精度が低く「？」なしに見えても実際のテロップには付いていることが多い

### ④ スライド上のテロップ・バナーは指摘不要
- スライドが表示されているシーンで、テロップやバナーが重なるのは正常な編集スタイル
- スライド上のバナー重なりは指摘しない

### ⑤ 句読点は指摘しない
- テロップに読点（、）や句点（。）がないのは**このチャンネルの正常スタイル**
- 「読点欠落」「句点なし」の指摘は出さない
- テロップに句読点が**入っている**場合（「。」「、」が不要に含まれている）は指摘する

### ⑥ チャンネル固有の用語は誤字扱いしない
- 「デカ重」など、チャンネルで使われている造語・スラングは誤字として指摘しない
- 不明な場合は「造語の可能性あり。演出意図の確認推奨」と添え書きする

## 5. 見逃しやすい項目（重点確認リスト）

### 最重要：見逃しが多いカテゴリ
1. **表記統一** — 「バイマ→BUYMA」のような揺れ。全フレームで一字一句確認
2. **語彙ミス** — 「バレー→バレエ」「スムース→スムーズ」など意味的な誤り
3. **数字範囲フォーマット** — 「40~50万円」→「40万~50万円」（万などの単位は各数字の後ろに付ける。円などの通貨単位は末尾1箇所のみでOK）
4. **助詞抜け** — 「クラファンオンラインで」→「クラファンをオンラインで」
5. **冗長語** — 「とかってなったら」「辞めてやめていくんで」「あと2つ目」
6. **送り仮名統一** — 「売上」vs「売り上げ」が動画内で混在していないか
7. **テロップ見切れ** — 画面端で文字が切れていないか（長文・英語・計算式は特に注意）
8. **テロップへの不要記号** — 「w」「ｗ」「！」「…」「。」「、」が不適切に入っていないか
9. **ひらがな優先語** — 「良い→いい」「位→くらい」「ご飯→ごはん」
10. **話途中カット** — 字幕が「〜ので」「〜て」「〜ですが」等で終わっているシーン

### 演出系の見逃し
- 演出シーンでバナーが非表示になっているか（先出しハイライト・迫力演出時は非表示が基本）
- 章タイトル・ポイント発表シーンでSE・演出テロップがあるか
- CTAセクションでQRバナーが毎回表示されているか（後半の再告知も含む）
- 複数箇条書きシーンで1項目ずつ小出しになっているか

### フレームだけでは確認不可（推測として指摘 or 「確認推奨」と明記）
- BGM音量・SE音量
- テロップ開始タイミングのズレ
- 色調・コントラスト（JPG圧縮で判断不可）
- 右上ロゴの素材種類
- テロップ・画像のスケール適正

## 6. ワークフロー

### Step 1: フレーム抽出
```bash
ffmpeg -i input.mp4 -vf fps=1 -q:v 2 frames/frame_%05d.jpg
```
- 1秒ごと1フレーム（fps=1）が基本。変更不可
- フレーム命名規則：`frame_00001.jpg`〜`frame_NNNNN.jpg`
- タイムコード変換：`frame_01346.jpg` → `1346 // 60 = 22分`, `1346 % 60 = 26秒` → `22:26`

### Step 2: 並列レビュー（3セクション分割）
- 動画を3等分して3つのエージェントが同時に担当
- 各エージェントは3〜5フレーム間隔でスキャン（これ以上広げると漏れが出る）
- 疑わしい箇所は前後フレームも確認

セクション分割例（1562フレームの場合）：
- Section 1: frame 1〜520（00:00〜08:40）
- Section 2: frame 521〜1041（08:41〜17:21）
- Section 3: frame 1042〜1562（17:22〜26:02）

### Step 3: script.txt の扱い
- `input/script.txt`（自動生成字幕）は参考程度
- 全文精読しない。固有名詞・ブランド名・通貨などをgrepで抽出するのみ
- script.txtは精度が低く信用できないため、フレーム目視が主たる判断根拠

### Step 4: レポート出力
```
output/backlog/{動画タイトル}/review_report.txt
```

フォーマット：
```
# {動画タイトル} 添削レポート

指摘件数: N 件

## HY1（誤字・固有名詞）
- 00:34 HY1 テロップ「xxx」→「yyy」に修正

## HY2（口語表現・語彙ミス）
- 03:29 HY2 テロップ「xxx」→「yyy」に修正

...
```

### Step 5: ファイル削除
添削完了後は必ず実施：
1. 動画ファイル（`tmp/{job_id}/input.mp4`）を削除
2. フレーム（`tmp/{job_id}/frames/`）を削除
- レポートのみ残す（動画1本あたり約154MB節約）

### Step 6: フィードバックの記録
ユーザーから誤検出・見逃しの指摘があった場合、および**添削以外の指示・ルール変更・フィードバックを受けた場合も**、`learnings/{REVIEWER_NAME}_learnings.md` に追記する。

```markdown
## 2026-03-27
- チャンネル固有の造語（「デカ重」等）は誤字扱いしない
- 「借して」→「貸して」の誤用は見逃しやすいため注意（HY1）
- タイムコードは必ずフレームファイル名から取得する。計算で出すとズレる
```

- `learnings/` フォルダが存在しない場合は自動作成する
- `.env` の `REVIEWER_NAME` が名前になる（未設定の場合は `unknown`）
- このファイルはfushimiが定期的に収集し、brainへの反映可否を判断する

## 7. スライド対応

スライドへの指摘が出た場合は、まずユーザーに確認する：
> 「このスライドは編集可能ですか？」

編集不可のスライドへの修正指摘は意味がないため、確認なしに指摘を出さない。

## 8. エンジン選択

### Claude（デフォルト・推奨）
- フレームを1枚ずつ読み込みながら添削（`read_frame` ツール使用）
- タイムコードはフレームファイル名から自動取得
- 4並列エージェントで処理
- ANTHROPIC_API_KEY が必要（Claude Code VSCode拡張経由で自動注入）

### Gemini（オプション・要明示指示）
- **ユーザーが「Gemini」と明示指示した場合のみ使用する。指示がない場合は必ずClaudeエンジンで処理する。**
- 動画ファイルをGemini File APIに直接アップロードして添削
- 55分以上の動画は前半/後半に分割 → 後半タイムコードにオフセット加算
- GEMINI_API_KEY が `.env` に必要
- モデル：`gemini-2.5-flash`（2M tokenコンテキスト版が必要な場合は `gemini-2.5-pro`）

## 9. よくある誤検出パターン（出さないこと）

| 誤検出パターン | 理由 |
|----------------|------|
| 読点（、）欠落 | このチャンネルは句読点なしが正常スタイル |
| バーコードMO指摘 | 指摘不要（編集D確認済み） |
| 実践者顔写真MO指摘 | 同意取得済み前提で指摘不要 |
| スライド上テロップ重なり | 正常な演出スタイル |
| 冒頭SEなし指摘 | フレームからは判定不可 |
| 「デカ重」等チャンネル固有語 | 造語・スラングは誤字扱いしない |
| フレーム未確認の疑問文「？」指摘 | script.txt精度が低く誤検出多い |
| 「より丁寧な表現に修正」系（具体的修正案なし） | 具体的な`→yyy`がない指摘は出さない |
